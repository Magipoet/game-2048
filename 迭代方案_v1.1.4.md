## 二、各版本详细迭代方案

### v1.0.5：计时器自动更新问题

#### 问题现象
- 计时模式的计时器不会自动显示计时过程（不会一秒一秒减少）
- 只有用户输入操作，计时器才会减少剩余时间
- 不限时模式的计时器也有相同问题
- 计时模式下即便设置了十分钟，只要用户不操作，游戏就不会结束

#### 问题原因

**问题代码：**

```dart
// 逻辑层 (game_2048_logic.dart) - 计时器更新但未通知 UI
void startTimer() {
  _gameTimer?.cancel();
  _elapsedSeconds = 0;
  _gameTimer = Timer.periodic(const Duration(seconds: 1), (timer) {
    _elapsedSeconds++;  // 数据更新了
    // 缺少：notifyListeners() 通知 UI
  });
}

// UI 层 (game_screen.dart) - 只在用户操作时刷新
Future<void> _handleMove(Direction direction) async {
  // ...
  if (result.isValid) {
    setState(() {});  // 只在用户移动时刷新
    _checkGameState();
  }
}
```

**原因分析：**
- 逻辑层：`Timer.periodic` 每秒更新 `_elapsedSeconds`，但没有通知 UI 层
- UI 层：`setState` 只在用户移动时调用，`_checkGameState` 也只在此时执行

#### 解决方案
采用 **ChangeNotifier + Listener** 模式：
1. 让 `Game2048Logic` 继承 `ChangeNotifier`
2. 在计时器更新时调用 `notifyListeners()` 通知 UI
3. UI 层添加监听器监听 `_gameLogic` 的变化，收到通知时刷新界面并检查游戏状态

### v1.0.7：获胜对话框和分数保存问题

#### 问题 1：获胜对话框重复弹出

**问题现象：**
- 用户合成 2048 后弹出"恭喜获胜"对话框
- 点击"继续"后一两秒，对话框再次弹出
- 重复这个过程，严重影响体验

**问题代码：**

```dart
// game_2048_logic.dart - isGameWon 不考虑继续游戏状态
bool get isGameWon => _gameWon;  // 只返回 _gameWon

// game_screen.dart - 检查逻辑
void _checkGameState() {
  if (_gameLogic.isGameWon && !_showWinDialog) {  // 问题：继续后 isGameWon 仍为 true
    _showWinDialog = true;
    _showWinGameDialog();
  }
}

void _continueGame() {
  setState(() {
    _showWinDialog = false;  // 重置为 false
    _gameLogic.continueAfterWin();  // 设置 _continueAfterWin = true
  });
}
```

**原因分析：**
- 调用链：计时器每秒触发 → 状态更新 → `_checkGameState()`
- `isGameWon` getter 只返回 `_gameWon`，不考虑 `_continueAfterWin`
- 用户点击"继续"后，`_showWinDialog` 被设为 `false`，但 `isGameWon` 仍为 `true`
- 下一秒条件再次满足，对话框重复弹出

**解决方案：**
新增 `shouldShowWinDialog` getter，条件为 `_gameWon && !_continueAfterWin`

#### 问题 2：手机端分数保存问题

**问题现象：**
- 手机端没有记录历史最高总分（电脑端有记录）

**问题代码：**

```dart
// game_screen.dart - 异步处理不当
void _checkGameState() {
  if (_gameLogic.shouldShowWinDialog && !_showWinDialog) {
    // ...
  } else if (_gameLogic.isGameOver && !_showGameOverDialog) {  // else if 结构有问题
    _showGameOverDialog = true;
    _updateHighScores();  // async 方法但没有 await
    // ...
  }
}

Future<void> _updateHighScores() async {
  await _storageService.setHighestScore(...);  // 异步保存
}
```

**原因分析：**
1. `_checkGameState()` 使用 `else if` 结构，增加逻辑复杂度
2. `_updateHighScores()` 是 async 方法但调用时没有 await
3. v1.0.5 之前计时器不自动更新，游戏可能不会"自然结束"

**解决方案：**
- 将 `else if` 改为两个独立的 `if` 语句
- 使用 `Future.then()` 确保异步操作完成后再显示对话框
- 添加 `mounted` 检查

### v1.0.8：棋盘大小和位置调整

#### 问题现象
- 棋盘几乎只在屏幕的上半部分
- 棋盘位置偏上，不符合真实使用场景
- 棋盘所占画面比例较小

#### 问题原因

**问题代码：**

```dart
// game_screen.dart - 棋盘占比和比例问题
Widget _buildPortraitLayout(double boardSize) {
  return Column(
    children: [
      Text('2048消消乐', style: TextStyle(fontSize: 24)),
      Expanded(
        flex: 3,  // 棋盘只占 3/7 ≈ 43%
        child: Center(child: BoardWidget(size: boardSize)),
      ),
      Expanded(
        flex: 4,  // 控制区占 4/7 ≈ 57%
        child: SingleChildScrollView(child: Column(...)),
      ),
    ],
  );
}

// 棋盘大小计算
double boardSize = minDimension * 5 / 6;  // 基于屏幕短边，竖屏时高度受限
```

**原因分析：**
- 棋盘大小使用 `minDimension * 5/6`，`minDimension` 是屏幕短边（宽度），导致棋盘高度受限
- 竖屏布局中棋盘 `flex: 3`，控制区 `flex: 4`，棋盘只占约 43% 空间
- 棋盘位置偏上，不符合手机握持时的操作习惯

#### 解决方案
1. **棋盘大小优化**：竖屏时使用屏幕高度的 70% 计算棋盘大小
2. **位置调整**：棋盘移至屏幕下方，符合握持操作习惯
3. **控制区布局**：控制区放在上方，模式按钮水平排列，当前得分和时间左右并排

### v1.1.1：修复布局崩溃问题

#### 问题现象
- 棋盘可以正常显示
- 游戏模块、得分等控制区组件全部消失
- 原位置变成灰色
- 无法进行任何操作

#### 问题原因

**问题代码：**

```dart
// mode_selector.dart - ToggleButtons 中错误使用 Expanded
Widget _buildHorizontalButtons() {
  return SizedBox(
    width: double.infinity,
    child: Row(              // ← 多余的 Row
      children: [
        Expanded(            // ← 在 Row 中的 Expanded 没问题...
          child: ToggleButtons(
            children: [
              Expanded(       // ← ToggleButtons 的 children 不能用 Expanded！
                child: Container(
                  child: const Text('限时 10 分钟'),
                ),
              ),
              Expanded(       // ← 这导致了布局约束冲突！
                child: Container(
                  child: const Text('不限时'),
                ),
              ),
            ],
          ),
        ),
      ],
    ),
  );
}
```

**原因分析：**
- `ToggleButtons` 内部会自动平分空间给它的 children，不需要用 `Expanded`
- 在 `ToggleButtons` 的 children 中使用 `Expanded` 会导致 **布局约束冲突**（无限约束循环）
- 多层嵌套的 `Row` + `Expanded` + `ToggleButtons` 导致整个 Widget 树无法正确计算尺寸
- 最终使控制区组件渲染失败，显示为灰色占位符

#### 解决方案
1. **移除多余嵌套**：删除 `Row` 和多余的 `Expanded` 层
2. **简化布局结构**：直接使用 `SizedBox(width: double.infinity)` 包裹 `ToggleButtons`
3. **正确使用 ToggleButtons**：children 中不再使用 `Expanded`，`ToggleButtons` 会自动处理布局

### v1.1.3：第一次滑动后才开始计时

#### 问题现象
- 一打开游戏界面就开始计时
- 用户还没开始操作，时间就已经在流逝
- 不符合真实使用场景

#### 问题原因

```dart
// game_screen.dart - 初始化时立即启动计时器
Future<void> _initServices() async {

    _gameLogic.startTimer();  // ← 问题：重置就计时

  });

```

**原因分析：**

3. **移除过早启动**：不再在初始化、重置、切换模式时调用 `startTimer()`
4. **重置标志**：游戏结束或重新开始时重置 `_timerStarted = false`

### v1.1.4：修复新数字生成位置问题和帮助按钮无响应问题

#### 问题 1：新数字生成位置错误

**问题现象：**
1. 滑动后新生成的数字可能出现在棋盘中间位置（如第三行第二列），而不是边缘
2. 滑动后新生成的数字没有出现在滑动方向的反向边缘：
   - 向上滑动时，新数字应该出现在最下方（第4行），但实际可能出现在左边或右边
   - 向左滑动时，新数字应该出现在最右侧（第4列）

**问题代码：**

```dart
// game_2048_logic.dart - 新数字生成逻辑
void _addRandomTile() {
  List<(int, int)> emptyCells = _board.getEmptyCells();
  if (emptyCells.isEmpty) return;

  Random random = Random();
  (int, int) position = emptyCells[random.nextInt(emptyCells.length)];
  // 从所有空单元格中完全随机选择，没有位置限制
}
```

**原因分析：**
- `_addRandomTile()` 方法从所有空单元格中完全随机选择位置
- 没有考虑滑动方向，无法判断应该在哪一侧生成新数字
- 不符合 2048 游戏的标准玩法（新数字应该出现在滑动方向的反向边缘）

**解决方案：**
1. **修改方法签名**：`_addRandomTile([Direction? direction])`，新增可选的方向参数
2. **传递方向信息**：在 `move()` 方法中调用 `_addRandomTile(direction)`，将滑动方向传递过去
3. **根据方向确定边缘**：
   - 向左滑动 → 新数字出现在最右侧（col = 3）
   - 向右滑动 → 新数字出现在最左侧（col = 0）
   - 向上滑动 → 新数字出现在最下方（row = 3）
   - 向下滑动 → 新数字出现在最上方（row = 0）
4. **三级回退机制**：
   1. 优先从目标反向边缘的空单元格中选择
   2. 如果目标边缘没有空位，则从所有四周边缘中选择
   3. 如果边缘都没有空位（极端情况），才从所有空单元格中选择

```dart
      child: IconButton(...),
    ),
    Padding(  // 第2层 - 游戏布局，覆盖在帮助按钮上面！
      padding: const EdgeInsets.all(16),
      child: ...,
    ),
    if (_showHelpDialog) _buildHelpDialog(),  // 第3层
  ],
);

// gesture_input_handler.dart - 点击事件拦截
return GestureDetector(
  // ...
  behavior: HitTestBehavior.opaque,  // 吸收所有点击事件，不传递给子组件
  child: child,
);
```

**原因分析：**
1. **Stack 层级问题**：在 Flutter 的 Stack 中，后面的组件会覆盖在前面的组件上面。游戏布局放在帮助按钮后面，导致游戏布局覆盖在帮助按钮上面，点击事件被游戏布局吸收
2. **HitTestBehavior 问题**：`GestureDetector` 使用 `HitTestBehavior.opaque` 会吸收所有点击事件，不传递给子组件

**解决方案：**
1. **调整 Stack 层级**：将帮助按钮移到游戏布局后面，确保它在最上层
2. **修改 HitTestBehavior**：将 `HitTestBehavior.opaque` 改为 `HitTestBehavior.translucent`，既能检测滑动手势，又能让子组件接收到点击事件

#### 问题 3：优化 CI/CD 构建流程

**问题现象：**
- GitHub Actions 运行后，Artifacts 区域会显示构建产物（Windows ZIP 和 Android APK）
- 这些 artifacts 会占用仓库空间，而实际上只需要在 Release 中保存即可

**问题原因：**
- 原 workflow 使用 `actions/upload-artifact` 将构建产物保存为 artifacts
- 然后在 `create-release` job 中下载 artifacts 并上传到 Release
- artifacts 会保留 90 天，占用不必要的存储空间

**解决方案：**
1. **移除 artifacts**：完全删除 `actions/upload-artifact` 和 `actions/download-artifact`
2. **直接上传到 Release**：
   - `build-windows` job 构建完成后直接创建 Release 并上传 Windows ZIP
   - `build-android` job 需要等待 build-windows 完成，然后向同一个 Release 追加 Android APK
3. **调整 job 依赖**：`build-android` 需要 `needs: [pre-check, build-windows]`
